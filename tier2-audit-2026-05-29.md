## Feature-by-Feature Audit

### #8 ‚Äî Preview Page Break Indicators

**Plan:** "Show faint dashed lines in the preview where PDF pages will split."

**Implementation:**
- `app.js:1235-1248` ‚Äî after `renderPreview()` finishes, measures `paper.scrollHeight`, divides by `792` (letter-page points), and inserts absolutely-positioned `div.page-break-indicator` elements.
- `style.css:1731-1739` ‚Äî 1px dashed gray line, `pointer-events: none`.

**Issues:**

1. **Bug: indicators won't anchor inside `.chart-paper`.**
   - Indicators use `position: absolute` with `left: 0; right: 0`. This requires the parent to have `position: relative`.
   - `.chart-paper` (`style.css:878-893`) does **not** set `position: relative`. The indicators will anchor to an arbitrary ancestor (or the viewport), rendering them offset from the paper content or completely outside it.
   - **Fix:** Add `position: relative;` to `.chart-paper`.

2. **Break positions will be wrong for most zoom levels.**
   - The math uses `pageHeight = 792` (points) while `.chart-paper` is `372px` wide in CSS pixels. 792 letter-points ‚âà 327 CSS pixels at paper scale, so the first break line draws ~145px too low even at 100% zoom. Breaks will appear to float between content blocks rather than on an actual page edge.
   - **Fix:** Compute the CSS-pixel equivalent of one letter page (accounting for paper width/padding ratio), and scale with current `previewZoom`. Example:
     ```js
     const pageHeightCSS = ((792 - 48) / 612) * paper.clientWidth; // approximate
     ```
   - Alternatively, render the breaks during PDF export only and hide them from the preview until this is reliable.

3. **Breaks are appended after `autoScaleLines` and `applyZoom`** ‚Äî fine, but they will accumulate duplicates if `renderPreview` is somehow called twice without clearing. Currently `paper.innerHTML = ''` at the start of `renderPreview` prevents this, so this is not a live bug, just a latent fragility.

**Verdict:** Present but visually broken ‚Äî a user enabling this feature would see indicators floating in the wrong places. Requires `position: relative` fix + a corrected pixel conversion before it's production-worthy.

### #10 ‚Äî Time Signature Field

**Plan:** "Add a `Time Sig` dropdown in the sidebar (4/4, 3/4, 6/8, etc.). Render it in the chart header next to BPM."

**Implementation:**
- `<select id="input-timesig">`: `index.html:102-108`. Five options: 4/4, 3/4, 6/8, 2/4, 12/8.
- State: `state.timeSignature`, default `'4/4'` (`app.js:118`).
- Sync: `app.js:327` (state‚Üêform), `app.js:338` (form‚Üêstate).
- Preview: `app.js:1043-1050` ‚Äî renders as `"{bpm} BPM ‚Ä¢ {timeSig}"`.
- PDF: `app.js:1516-1529` ‚Äî same combined string at 12px italic.

**Issues:**

1. **Default `4/4` is always rendered in preview when BPM is set**, even if the user never touched the time-sig dropdown. This is because `createEmptyChart()` hardcodes `'4/4'` and the render condition is `if (state.bpm || state.timeSignature)`.
   - Not strictly wrong (most songs are 4/4), but a chart with only a BPM and no explicit time signature will still show "120 BPM ‚Ä¢ 4/4", which is slightly misleading.
   - **Polish:** Treat `'4/4'` as "unset" ‚Äî only render it if the user explicitly chose a non-default value, or default to `''` in `createEmptyChart()`.

2. **Minor layout concern:** BPM + time sig are joined with ` ‚Ä¢ ` which may wrap on narrow previews if BPM is 3 digits. Not a real-world problem at 372px paper width.

**Verdict:** Fully functional. The default-value display quirk is cosmetic and non-blocking.

### #12 ‚Äî Dark Mode Preview Toggle

**Plan:** "A toggle in the preview header to switch the paper between white (print-accurate) and a dark/inverted reading mode."

**Implementation:**
- Button: `index.html:245`, üåô emoji, title "Toggle Dark Mode".
- JS handler: `app.js:2220-2225` ‚Äî `chartPaper.classList.toggle('dark-mode')`.
- CSS rule: `style.css:1741` ‚Äî `#chart-preview-paper.dark-mode { filter: invert(1) hue-rotate(180deg); }`.

**üî¥ Critical bug ‚Äî dark mode does not work:**

The CSS selector is `#chart-preview-paper.dark-mode`, but the HTML element's id is `chart-paper` (`index.html:253`). The JS toggles the `dark-mode` class on `#chart-paper`. **The CSS rule never matches, so the filter is never applied.** Clicking the üåô button does nothing visually.

**Fix:** Change `style.css:1741`:
```css
/* was: */
#chart-preview-paper.dark-mode { ... }
/* to: */
#chart-paper.dark-mode { ... }
/* or, better and more robust: */
.chart-paper.dark-mode { ... }
```

**Additional polish:**
- No visual affordance that dark mode is "on" ‚Äî the üåô button stays the same. Consider toggling to ‚òÄÔ∏è or adding an `aria-pressed` / active class.
- `filter: invert(1) hue-rotate(180deg)` is a reasonable approximation but will distort chord colors (blues become oranges). Worth mentioning in the UI or offering a dedicated dark theme rather than relying on CSS filter.

**Verdict:** Completely broken in a visible way ‚Äî a one-line CSS selector fix. Must fix before shipping.

### #13 ‚Äî Improved Saved Charts Library

**Plan:** "Show key, section count, date. Sort options. Search/filter. Star/pin favorites."

**Implementation:**
- Search input `#library-search`: `index.html:177`.
- Sort dropdown `#library-sort`: `index.html:178-182` ‚Äî options: Newest, A-Z, Key.
- Entry metadata: `key`, `sectionsCount`, `savedAt`, `isFavorite` persisted on each chart entry (`app.js:262-270`).
- Sort logic: `app.js:1875-1881` ‚Äî favorites bubble to top, then sort by chosen criterion.
- Favorites toggle UI: `app.js:1916-1927` ‚Äî ‚òÖ/‚òÜ button, persists to localStorage immediately.
- Meta display: `app.js:1909-1913` ‚Äî "Key: G ‚Ä¢ 8 sections ‚Ä¢ 5/29/2026".
- CSS: `style.css:1749-1791` ‚Äî `.library-item`, `.library-item-title`, `.library-item-meta`, `.favorite-btn`.

**Issues:**

1. **Migration gap: old charts pre-Tier 2 lack `data.id`.**
   - `saveChartToLibrary` (`app.js:257-260`) looks up existing entries by `c.data.id === state.id`. Charts saved before Tier 2 had no `state.id` field, so `findIndex` returns `-1` ‚Üí duplicate entry is created instead of updating the original.
   - `loadChartFromLibrary` (`app.js:289`) patches in `state.id` on load but never persists it back ‚Äî so on subsequent saves the duplicate behavior repeats.
   - **Fix:** In `saveChartToLibrary`, fall back to name-based dedup if `data.id` is missing:
     ```js
     const existing = charts.findIndex(c =>
       (c.data.id && c.data.id === state.id) ||
       (!c.data.id && c.name === state.title)
     );
     ```
     This preserves the old name-keying behavior for legacy entries.

2. **Favorites are persisted correctly** ‚Äî the `isFavorite` field is carried forward from the existing entry on re-save (`app.js:268`). ‚úÖ

3. **Sort by Key** works via locale compare on `a.key || ''`. Charts without a key sort before those with one ‚Äî acceptable. ‚úÖ

4. **Search query is applied after sort.** The filtered list still sorts favorites-first. ‚úÖ

**Verdict:** Feature-complete and polished. The legacy-chart migration gap (#1) is a real bug for existing users ‚Äî they'll see their charts duplicated when resaved. Fix is small and backwards-compatible.
